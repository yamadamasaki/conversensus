# Conversensus Operation Manual in development

このドキュメントは, プロダクト・オーナが開発中の Conversensus の動作確認を行うための操作マニュアルである.

このドキュメントは, 将来的にはユーザーズ・マニュアルや, PRD のベースになることを想定している.

## 起動

### ATProto 向け開発時環境

#### PDS の起動

- `bash infra/pds/setup.sh` # 環境変数を `infra/pds/.env` に設定する. 既に設定済みならば不要
- Docker Desktop などで docker daemon を起動しておく
- `cd infra/pds && docker compose up -d` # container で PDS を起動する
- `curl http://localhost:2583/xrpc/com.atproto.server.describeServer` # 起動を確認する

#### アカウントの作成

あらかじめ, テスト用のアカウントを作成する.

まずは招待コードを作成する.

```shell
curl -X POST http://localhost:2583/xrpc/com.atproto.server.createInviteCode \
    -H "Content-Type: application/json" \
    -H "Authorization: Basic $(echo -n 'admin:7147f1f843bbf2a2f97b69a30d797543' | base64)" \
    -d '{"useCount": 1}'
```

招待コードが返ってくる.

```json
{"code":"localhost-lvkjj-wiq6n"}
```

それを使って, アカウントを作成する.

```shell
curl -X POST http://localhost:2583/xrpc/com.atproto.server.createAccount \
    -H "Content-Type: application/json" \
    -d '{
      "email": "masaki@metabolics.co.jp",
      "handle": "alice.test",
      "password": "devpassword123",
      "inviteCode": "localhost-lvkjj-wiq6n"
    }'
```

いったん, アカウントを作成したら, web クライアントは `/client/src/.env.local` (dev 環境時) の `VITE_ATPROTO_{HANDLE,PASSWORD}` を参照して自動的にログインする.

[未実装] ログアウト機能は今はまだないので, そのためには cookie を削除する必要がある.

[未実装] どのアカウントで入っているのか, handle を表示した方が良い

[未実装] 一つのクライアントのコード・ベースが一つのアカウントに独占されてしまうので, なんとかしたい

#### server/client の起動

- `bun run dev:server`
- `bun run dev:client`

vite (7) は node 20.19+ or 22.12+ を要求する. v24 を入れた.

http://localhost:5173/ で client が走っているはず.

## グラフ管理

グラフ管理の対象は, File, Sheet である. Branch については後述する.

### 新規ファイル作成

左サイドバーの一番上の input フィールドにファイル名を入れ, 右隣の '+' ボタンをクリックすることによって, 新しいファイル (このファイルは OS のファイル・システム上のファイルではなく, このシステム内で定義されたもの) が作られる.

既存のファイルと同名のファイルを作ることもできる. それらは内部 ID で区別されている.

input フィールドの左端の '↑' ボタンをクリックすることによって, export した .conversensus ファイル (後述) を import できる.

### ファイル管理

その下には, 既に作られたファイルの一覧がある.

行の左端のアイコンをクリックすると, そのファイルに含まれるシート (後述) の一覧が展開して, 表示される.

行の右端の「設定」アイコンをクリックすると, ダイアログが表示され, その中で以下の操作ができる.

- ファイル名の変更
- ファイルの概要の入力
- ファイルの export (.conversensus という拡張子を持つ JSON ファイル)
- ファイルの削除
- 名前や概要の変更の保存
  - ESC キーを押したり, ダイアログ外をクリックすると保存せずにダイアログを閉じる

### シート管理

シートは, 一枚の (大きな) グラフである. 一つのファイルには, 複数のシートが収められる.

ファイルの内容一覧の一番下の '+ シートを追加' をクリックすると, 新しいシートが追加される. その時点でのシートの名前は "Sheet"+数字 というようなものであるが, 後で変更することもできる.

シート一覧の各行の右端の「設定」アイコンをクリックすると, ダイアログが表示され, その中で以下の操作ができる.

- シート名の変更
- シートの概要の入力
- シートの削除
- 名前や概要の変更の保存
  - ESC キーを押したり, ダイアログ外をクリックすると保存せずにダイアログを閉じる

## グラフ・エディタ

グラフ・エディタの対象は, Node, Edge, Group である.

### ノード

- ノード作成
  - グラフ画面の何もないところをダブルクリックすると, そこにノードが追加される
- ノード内容の編集
  - ノードの内部をダブルクリックすると, そのノードの内容を編集できるようになる. 内容はテキストであるが, markdown 形式のマークアップをサポートする. サポートされるマークアップは [react-markup](https://remarkjs.github.io/react-markdown/) が提供するものである (ただし, 画像の埋め込みはサポートされない)
  - 編集を確定するには, 編集エリアの外をクリックする
  - ESC キーを押すと編集はキャンセルされる
  - 編集中はテキスト入力に対する undo/redo が可能である
- ノード移動
  - ノードの内部をドラッグすると, そのノードを移動できる
- ノードの大きさ変更
  - ノードの上下左右辺あるいは四隅のハンドルをドラッグすると, そのノードの大きさを変更できる
- ノード削除
  - ノードを選択して delete キーを押すと, そのノードが削除される. ノードが削除されると, そのノードに接続しているエッジも削除される
- ノードの選択
  - ノードをクリックすると, そのノードが選択される. 選択されたノードは, 青い境界線で表示される
  - 続いて別のノードをクリックすると, そのノードが選択され, 先に選択されていたノードの選択は解除される
  - command + クリック (Mac) / control + クリック (Windows) すると, 選択されたノードに追加される
- ノードの copy/paste
  - ノードを選択して command + c (Mac) / control + c (Windows) を押すと, そのノードがコピーされる
  - command + v (Mac) / control + v (Windows) を押すと, コピーされたノードがペーストされる. ペーストされたノードは, コピーされたノードと同じ内容を持ち, かつ, コピーされたノードの右下に配置される
- [未実装] ノードのラベル編集
- [未実装] プロパティ

### エッジ

- エッジ作成
  - ノードの上下左右にあるハンドル (黒丸) をドラッグして, もう一方のノードのハンドルにドロップすると, その二つのノードを接続するエッジが作成される
- エッジのラベル編集
  - エッジをダブルクリックすると, そのエッジのラベルを編集できるようになる
  - 編集を確定するには, ENTER キーを押すか, 編集エリアの外をクリックする
  - ESC キーを押すと編集はキャンセルされる
  - 編集中はテキスト入力に対する undo/redo が可能である
- エッジの種類の変更
  - エッジを右クリックすると現れるコンテキストメニューから, そのエッジの外見の種類を変更できる. 現在は以下の種類がある
    - Bezier
    - Straight
    - Step
    - Smooth Step
- エッジ削除
  - エッジを選択して delete キーを押すと, そのエッジが削除される

### グループ化

- グループ作成
  - 一つ以上のノードを選択して, command + g (Mac) / control + g (Windows) を押す, あるいは右上の「グループ化」ボタンをクリックすると, それらのノードをグループ化できる
    - 選択されたノードは, 青色の境界線で表示される
  - 作成されたグループは, それらのノードを囲むような青い背景と青い太めの境界を持つ矩形で表示される
  - グループの上部にはラベル (デフォルトで "グループ") が表示される. ラベルをダブルクリックすると, グループのラベルを編集できるようになる
- グループ移動
  - グループの内部をドラッグすると, そのグループを移動できる. グループを移動すると, そのグループに含まれるノード (とエッジ) も一緒に移動する
- グループの大きさ変更
  - グループの上下左右辺あるいは四隅のハンドルをドラッグすると, そのグループの大きさを変更できる
    - その結果, ノードがグループの外にはみ出てしまうことがある. その場合にはノードはグループに含まれたままだが, 見た目上はグループの外に出ているように見える
  - ノードを移動したり, 大きさを変更したりして, グループの外にはみ出た場合には, グループの大きさを自動的に変更して, そのノードをグループの中に収める
    - [未完成] いったんグループの大きさを変更すると, この機能はオフになる
- グループ解除
  - グループを選択して delete キーを押すと, そのグループが削除される. グループが削除されても, そのグループに含まれていたノードやエッジは削除されない
  - [未実装] ノードのグループへの出し入れをノードの drag & drop でできる. あるいは cut/copy/paste でもよい (グループが選択されている場合には, そのグループのメンバとして paste される)
    - 未実装なので, 今はいったんグループを解除して, 再度グループ化する必要がある
- グループの copy/paste
  - グループを選択して command + c (Mac) / control + c (Windows) を押すと, そのグループがコピーされる
  - command + v (Mac) / control + v (Windows) を押すと, コピーされたグループがペーストされる. ペーストされたグループは, コピーされたグループと同じ内容を持ち, かつ, コピーされたグループの右下に配置される

## バージョニング

シート毎に, ブランチを作成して複数のバージョンを管理する.

ブランチは, それぞれそのバージョンのシートの内容と変更履歴を保存し, 承認されれば trunk (main) に統合 (merge) される.

### ブランチの作成と利用

- ブランチ作成
  - 左サイドバーのシート名の下の "+ branch" をクリックすると, その時点のシートの内容を基点として, 新しいブランチが作成される
- ブランチ内での編集
  - あるブランチに切り替えた状態でシートを編集すると, その編集はそのブランチの変更履歴に保存される
  - 基点となっているシートの内容との差異は, 以下のように表示される
    - 追加: オレンジ色 (#f97316) のハイライト
    - 変更: グリーン (#22c55e) のハイライト (以前は実装されていたはずだが, 現在は機能していない [バグ])
    - 削除: [未定義]
- ブランチ・コミット
  - ブランチを切り替えた状態でシートを編集しているときに, シート右下の "コミット" ボタンをクリックすると, そのブランチに対する変更がコミットとして保存される
    - そのときに変更理由も記述できる
- ブランチ切り替え
  - 左サイドバーのシート名の下にあるブランチの一覧から, ブランチをクリックすると, そのブランチに切り替わる
  - あるいはシート名自体をクリックすると, trunk (main) に切り替わる
  - いずれの場合も切り替え前のブランチの編集内容は保存される
- ブランチの統合
  - ブランチをコミットすると, そのブランチの今の状態を trunk に統合することができるようになる
  - "merge" ボタンをクリックすると, そのブランチの内容が trunk に統合される
    - そのときに統合理由も記述できる
    - [未実装] 現状では merge はそのブランチによる trunk の上書きになるが, 本来は 両者のイベント列の merge になる
      - つまり, ブランチを切った後に trunk に加えられた変更も, ブランチの変更とともに trunk に統合されるようになる
- ブランチのクローズ
  - merge されていないコミットが存在していない状態では, そのブランチを close して, それ以上の変更ができないようにできる
- ブランチ削除
  - ブランチ自体の存在を忘れるためには, 未 commit であっても 未 merge であっても, ブランチを削除することができる
  - 削除されたブランチには, 今後アクセスすることはできない
- [未実装] ブランチのブランチ
