# ATProto 統合戦略

> 作成日: 2026-03-31
> セッション: S225〜S228 での議論に基づく

---

## 背景・意図

conversensus は, 将来の分散型ソフトウェアエコシステムに向けたプラットフォームとして構想されている. 大企業による中央集権的制御から, 小規模コミュニティのフェデレーションへの移行 (約10年先のホライゾン) を見据え, 合意形成ツールを分散インフラ上で動かすことが目標である.

**核心的な問い:** ユーザーが同じノードを同時に編集した際の「コンフリクト」を, バグとして隠蔽するのではなく, 合意形成プロセスの一部として可視化できるか?

---

## プラットフォーム選定

### 評価した選択肢

| プラットフォーム | 評価 |
|---|---|
| **Tim Berners-Lee's SOLID** | 技術的には優れるが, エコシステムが小さく実験フェーズには不向き |
| **ATProto** | 現時点での実験に最適. フェデレーション設計が conversensus の思想に合致 |
| **Internet Computer Protocol** | ブロックチェーン依存が重く, 現実的な採用コストが高い |

**決定: ATProto を採用**

---

## ATProto の特性と conversensus との整合

### ATProto MST の役割

- ATProto の PDS (Personal Data Server) は **Merkle Search Tree (MST)** をリポジトリデータ構造として義務付けている
- MST は「CRDT」ではなく, **コンテンツアドレス型の追記専用ストア**である
  - 目的: 暗号的検証と効率的な同期
  - OR-Set セマンティクス: レコードの追加・削除はコンフリクトフリー
  - **ただし**: 同一エンティティへの並行プロパティ変更は解決しない

### conversensus の LPG マージ戦略との比較

| 観点 | ATProto MST | conversensus LPG マージ |
|---|---|---|
| 粒度 | レコード単位 (追加/削除) | エンティティ内のプロパティ変更まで |
| コンフリクト解決 | OR-Set で add/delete はフリー | ラベル付き有向グラフのマージ |
| 設計思想 | 検証・同期の効率化 | 合意形成プロセスの可視化 |

**重要な洞察:** ATProto が解決しない「同じノードラベルへの並行変更」こそ, conversensus が可視化すべき合意形成の機会である.

## アーキテクチャ設計

### 2層構造

```
[ATProto 層]  追記専用レコードストア (検証・同期)
      ↓
[App View 層] コンフリクト可視化 + 合意形成 UI
```

**ATProto 層の役割:**
- ノード・エッジを追記専用レコードとして保存
- 各変更が新レコードを生成 (イミュータブル)
- OR-Set セマンティクスで add/delete はコンフリクトフリー

**App View 層の役割:**
- エンティティレベルのコンフリクトを検出
- コンフリクトをグラフ上に可視化 (「対立ノード」として表示)
- LWW (last-write-wins) または明示的な合意で解決

### Lexicon 設計方針 (草案)

セマンティックデータとレイアウト情報を **ATProto レコードレベルで分離:**

```
app.conversensus.node       ← セマンティック (id, content, parentId)
app.conversensus.edge       ← セマンティック (source, target, label)
app.conversensus.nodeLayout ← レイアウト (nodeId, x, y, width, height)
```

この分離により:
- グラフ構造 (セマンティック) は全ユーザーで共有・同期
- レイアウト (表示位置) はユーザーごとに個別管理可能
- コンフリクトの発生源をセマンティック変更に限定できる

## 現在のコードベースとの対応

### 実施済みの対応 (2026-03-31)

スキーマ v2 リファクタリングで ATProto Lexicon 設計に向けた準備完了:

- `GraphNode` から `style` プロパティを削除 → セマンティックのみに
- `NodeLayout` を独立エンティティとして新設 (`nodeId` 参照で紐付け)
- `Sheet.layouts[]` 配列でレイアウトを分離管理
- v1→v2 マイグレーション関数 (`migrateV1toV2`) で後方互換性を維持
- ファイルバージョンを `'1'` → `'2'` に更新

### コードベースの事前対応 (変更前から存在)

- イベントカテゴリが既に `structure / content / layout / presentation` に分類済み
- `NODE_MOVED`, `NODE_RESIZED` は `layout` カテゴリ
- イベントソーシング + undo/redo スタックが分散編集シナリオに親和性が高い

## 良い点

1. **思想の一致:** ATProto の分散・フェデレーション設計が conversensus の「小コミュニティのネットワーク」ビジョンに直接対応
2. **既存エコシステム:** Bluesky により実証済み. PDS インフラが存在する
3. **追記専用との相性:** コンフリクトを「上書き」ではなく「並行提案」として保存できる
4. **OR-Set で構造変更はフリー:** ノード追加・削除のコンフリクトは自動解決

## 懸念点・制約

1. **プロパティ変更コンフリクトは未解決:** 同一ノードの label 同時編集は ATProto 側では解決されない → App View で対処必要
2. **Lexicon の標準化が未定:** `app.conversensus.*` スキーマの設計はこれから
3. **ATProto 認証・PDS 接続:** OAuth / DID 認証の実装コストが高い
4. **現在はシングルユーザー前提:** サーバーに排他制御なし, 並行編集のインフラがない

---

## 当面の方針 (フェーズ分け)

### Phase 1: スキーマ設計 (現在)

- セマンティック / レイアウト分離 (✅ v2 スキーマ実装済み)
- ATProto Lexicon 草案の設計
- 将来の分散対応を考慮したデータモデルの確定

### Phase 2: ATProto インフラ統合 (次のフェーズ)

- ATProto PDS への接続
- DID ベースのユーザー認証
- レコードの読み書き実装

### Phase 3: App View とコンフリクト可視化

- 並行変更の検出と可視化
- 合意形成 UI の実装
- LWW または明示的投票による解決

> **原則:** セマンティックスキーマを先に安定させ, インフラ統合は後で.
> 設計の見通しなしに ATProto 接続を始めない.
