# SyncStatusIndicator テスト仕様

## 何を

`SyncStatusIndicator` (step1 W3d5-6) をテストする。remote 未送信件数の表示 (0 件 / N 件 /
上限超過時の「N 件以上」)、未ログイン時の非表示、「今すぐ同期」による手動 flush と、失敗時に
件数が残って再送できることを検証する。

## なぜ

remote への送信は best-effort・非ブロッキング (§3.1) なので、**失敗しても編集は何事もなく続く**
= ユーザは自力では気づけない。この表示と手動再送が、設計 §2 の目標「同期失敗にユーザが気づき、
回復できる」の UI 側の担保そのもので、ここが壊れると純 fire-and-forget と実質同じになる:

- **未ログイン時の非表示 (§3.7)**: remote 経路が無いのに同期概念を見せると、ローカルのみで
  使っているユーザに無意味な不安を与える。`remoteQueue=null` で完全に何も描かないことを固定する。
- **件数の追従**: `RemoteSyncQueue.subscribe` の「登録直後に現在値 + 以後の更新」契約に依存して
  いる。マウント後に積まれた分が表示に出ることを固定する (購読を張り忘れても初期表示だけは
  正しく見えてしまうため、**後から enqueue** して確かめる)。
- **上限の頭打ち表現 (D1)**: 上限に達すると実際の未送信はもっとある。「2 件」と言い切ると嘘に
  なるので「2 件以上」と見せる。溢れた分はローカル正典に残り起動時 catch-up で回収される。
- **手動回復 (§3.7)**: オフライン→復帰の自動検知 (W3d5-7) を待たずユーザが能動的に直せること。
  **失敗しても件数が残る** (破棄しない) こと、復帰後の再クリックで 0 件に戻ることを固定する。

## どのように

`@testing-library/react` の `render` + `fireEvent` で描画・クリックする。キューは実物の
`RemoteSyncQueue` に `FakeProvider` (`online` で push の成否を切替) を包んで注入し、フィルタ・
保持・再送の実挙動を通す (PDS 非依存)。enqueue とクリックは購読通知と flush で state 更新を
起こすので `act` で包む。

- `remoteQueue=null` → `container.textContent` が空 (何も描画しない)。
- 0 件 → 「クラウド同期済み」、ボタンなし。
- マウント後に 2 件 enqueue → 「クラウド未同期: 2 件」。
- `capacity: 2` で 3 件 enqueue → 「クラウド未同期: 2 件以上」。
- 「今すぐ同期」クリック → provider に push され「クラウド同期済み」へ戻る。
- offline でクリック → 「クラウド未同期: 1 件」のまま (破棄しない)。online に戻して再クリック
  → 「クラウド同期済み」。
