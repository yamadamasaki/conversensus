# actor テスト仕様

## 何を

`sync/actor.ts` (step1 Phase 4d-2) をテストする。この端末の一意 id (`getDeviceId`) と、
DID と組み合わせて Lamport の操作主体を作る `composeActor` の 2 つ。

`localStorage` はテスト毎に新しい fake (Map ベース) を渡し、状態を分離する。

## なぜ

**受信 (Phase 4d) には「因果順序の単位と重複排除の単位を識別できる」ことが要る。**

W3d5-7 の実機検証で、device A・B とも `actor = 'local'` を名乗り、両端末の batch が
ともに `clock=3` で PDS に載った。これは偶然ではなく構造的な帰結で、同一 snapshot から
genesis した端末は同じ clock から発番を始める (設計 `step1-phase4d-receive.md` §1.1)。
actor が端末を区別しない限り、受信側は 2 つの batch を因果的に順序づけることも、
同一操作かどうか判定することもできない。

形式を `<did>#<deviceId>` の複合にしたのは、片方だけでは不足するため:
- **DID だけでは不可** — 想定アクターモデルが「単一ユーザー・複数端末」なので、同じ
  アカウントの別端末を区別できない。
- **deviceId だけでも不可** — 出自 (誰の編集か) が失われ、権限・帰属・PDS リポジトリ
  所有者との照合ができない。step2 の共同編集で必要になる。

## どのように

- **getDeviceId**:
  - 初回は生成して保存する (保存されたことをストレージ経由で確認)。
  - 二度目以降は保存済みの値を返す — **起動をまたいで安定**であることが本質。
  - 既存値があれば尊重し再生成しない。
  - 別ストレージ (= 別端末) なら別 id になる。
  - `localStorage` がクリアされたら新しい id になり、その端末は「別の端末」として振る舞う。
    **正しさは失われない** (actor が 1 つ増えるだけ) ので、再生成を防ぐ仕組みは持たず、
    したがってそれを固定するテストも置かない。
- **composeActor**:
  - ログイン中は `<did>#<deviceId>`、未ログインは `local#<deviceId>`。
  - **同じユーザーでも端末が違えば別 actor** — 4d-2 の要点そのもの。上記 W3d5-7 の
    実測が再発しないことを表す。
  - 同じ端末でもユーザーが違えば別 actor。
  - 同じ端末・同じユーザーなら安定している (受信側が識別の根拠にできる)。

**人間可読な値 (端末名など) を混ぜない**方針はテストでは固定していない — actor は PDS 上の
レコードに載って公開されるため実装コメントで戒めているが、「混ぜていないこと」を機械的に
検査する手段がないため。
